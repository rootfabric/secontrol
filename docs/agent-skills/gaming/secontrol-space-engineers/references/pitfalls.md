# Подводные камни secontrol — Полный справочник

Все подводные камни и неочевидные моменты при работе с библиотекой secontrol и Space Engineers.

> **Краткая сводка**: в [SKILL.md](../SKILL.md) есть топ-10 самых критичных подводных камней. Ниже — полный список.

## Сетки (Grids)

- **`Grid.from_name()` делает нечёткий поиск** и возвращает **первое совпадение**. Если несколько сеток имеют похожие имена (например "DroneBase" и "DroneBase 2"), вызов `Grid.from_name("DroneBase")` вернёт "DroneBase 2". **Всегда используйте `get_all_grids()` для получения списка сеток с их ID, затем создавайте `Grid` напрямую с явным `grid_id`** или используйте полное точное имя (например `"DroneBase 2"`).
- **`prepare_grid()` ТРЕБУЕТ аргумент STRING, не int!** Передача целого числа вроде `prepare_grid(76365856444180915)` интерпретирует его как `existing_client` (первый позиционный аргумент), а НЕ как `grid_id`. Функция тогда разрешает сетку по умолчанию (часто неправильную!). Все последующие операции молча работают с неправильной сеткой. Правильно: `prepare_grid("76365856444180915")` (строка) или `prepare_grid("skynet-baza2")`. Всегда проверяйте, что вывод "Resolved grid" соответствует ожиданиям.
- **`prepare_grid()` вызывает `ValueError: Multiple grids found` при дублирующихся именах.** Если две сетки имеют одинаковое имя (например две копии "skynet-baza0"), `prepare_grid('skynet-baza0')` завершается ошибкой. Исправление: используйте ID сетки напрямую: `prepare_grid('127817843801970018')`. Проверьте доступные сетки с помощью `get_all_grids()`.
- **ID сеток могут меняться между сессиями.** Перезапуски сервера SE могут переназначать ID сеток (например skynet-baza0 сменился с 118168110731275470 на 91207270182100228). Всегда используйте `get_all_grids()` для поиска текущих ID, а не хардкод из предыдущих сессий.
- **`Grid` не имеет атрибута `.id` — используйте `.grid_id`.** Также `.name`, `.grid_key`, `.owner_id`, `.player_id`, `.metadata` (сырой dict из Redis). Список `.metadata['blocks']` содержит полные данные блоков (type, subtype, id, state, local_pos, bounding_box, mass) как сырые dict — полезный запасной вариант, когда `.blocks` возвращает неполные данные или только ID.
- **Конструктор `Grid()` требует 3 позиционных аргумента: `owner_id`, `grid_id`, `player_id`.** Никогда не вызывайте `Grid()` без аргументов — используйте `prepare_grid("grid_name")` или `Grid.from_name("grid_name")` для поиска по имени, или создавайте явно: `Grid(redis, owner_id, grid_id, player_id, grid_name)`.
- **`grid.blocks` может вести себя по-разному в зависимости от состояния сетки.** На некоторых сетках итерация `grid.blocks` возвращает только ID блоков (целые числа), тогда как на других — пары `(block_id, BlockInfo)`. В случае сомнений обращайтесь к деталям блоков через `grid.metadata['blocks']`, который всегда возвращает полный сырой dict для каждого блока.
- **Инвентарь устройств сетки может меняться между сессиями.** Блоки могут быть удалены, разрушены или уничтожены в игре. `get_first_device(OreDetectorDevice)` может вернуть `None`, даже если детектор был в предыдущей сессии. Всегда проверяйте на `None` и сообщайте пользователю, какие сетки в данный момент имеют нужное устройство. Используйте `grid.blocks` с `block_type='MyObjectBuilder_OreDetector'` для подтверждения.

## Remote Control (RC)

- **`rc.enable()` включает АВТОПИЛОТ, а НЕ блок.** `rc.enable()` отправляет команду `autopilot_enable` — включает систему автопилота, а не сам блок RC. Для включения блока (питание вкл/выкл) используйте `rc.set_enabled(True)`. Однако некоторые блоки (RC, Cockpit, Conveyors) **не могут быть включены через API** — `set_enabled(True)` возвращает 1, но телеметрия по-прежнему показывает `enabled: False`. Их нужно включать в игре через терминал. Всегда проверяйте `rc.telemetry.get('enabled')` после вызова `set_enabled()`.
- **RC `enabled: False` не всегда означает поломку.** Блок RC может сообщать `enabled: False`, но по-прежнему отвечать на команды `goto()` — система автопилота работает независимо от состояния питания блока. Проверьте с помощью `rc.goto(small_target, speed=5)` и проверьте, меняется ли скорость.
- **Автопилот RC перелетает цель в космосе.** Автопилот не может полностью остановить корабль из-за инерции. Когда расстояние < 40м, отключите RC (`rc.disable()`), снова включите с демпферами (`rc.enable(); rc.dampeners_on()`) и ждите пока скорость < 0.3 м/с. Без этого корабль пролетает мимо цели на скорости 2-3 м/с.
- **RC goto() включает автопилот асинхронно.** `rc.goto()` возвращает немедленно (result=1), но автопилоту требуется 1-3 секунды для включения. Проверяйте `rc.telemetry['autopilotEnabled']` в цикле опроса. Автопилот также может самопроизвольно отключиться через несколько секунд, если блок RC в деградированном состоянии.
- **Избежание столкновений блокирует стыковку с базами на астероидах.** Встроенное SE `set_collision_avoidance(True)` обнаруживает воксели астероидов рядом с базой и останавливает корабль в 30-50м. **Всегда отключайте избежание столкновений для стыковки с базой** и обрабатывайте препятствия вручную или принимайте риск. Коннектор базы находится НА поверхности астероида — корабль должен пролететь через "опасную зону" для стыковки.
- **Требуется пилот или RC для работы двигателей.** Телеметрия кокпита `hasPilot: False` + нет `RemoteControlDevice` на сетке = все команды `set_thrust(override=N)` молча игнорируются. Корабль выглядит так, будто двигатели работают (блоки включены, команды возвращают `sent=1`), но не производит ускорения. Проверяйте: `cockpit.telemetry.get('hasPilot')` или `cockpit.telemetry.get('isUnderControl')`. Исправление: (а) посадите пилота в кокпит в игре, или (б) добавьте блок Remote Control.
- **CockpitDevice предоставляет телеметрию полёта без RemoteControl.** `CockpitDevice` предоставляет ту же телеметрию, что и `RemoteControlDevice`: `position`, `orientation`, `linearVelocity`, `angularVelocity`, `speed`, `gravity`, `shipMass`, `dampenersOverride`. Используйте `get_orientation(cockpit)` и `get_world_position(cockpit)` — они принимают любое `BaseDevice`. Метод `set_dampeners(enabled: bool)` — правильный API.
- **`CockpitDevice.set_dampeners()` принимает bool, а не dict.** Правильный вызов: `cockpit.set_dampeners(True)` или `cockpit.set_dampeners(False)`.

## Устройства и блоки

- **Новообретённые/спроецированные корабли имеют большинство блоков отключёнными.** После проекции + сварки только батареи, солнечные панели и гироскопы обычно включены. Все двигатели, конвейеры, RC, кокпит и функциональные блоки начинают отключёнными. Используйте паттерн "включить все блоки" перед любой автоматизацией. Не все блоки могут быть включены через API.
- **Не все блоки могут быть включены/отключены через API.** Мод SE обрабатывает команды включения/только для определённых типов блоков:
  - ✅ **Можно включить**: ConnectorDevice, ThrusterDevice, BatteryDevice, GyroDevice, OreDetectorDevice, ShipWelderDevice, NanobotDrillSystemDevice, RefineryDevice, AssemblerDevice, GasGeneratorDevice, HydrogenEngine (GenericDevice), SolarPanel (GenericDevice)
  - ❌ **Нельзя включить**: LargeShipMergeBlock, LargeBlockLargeContainer (CargoContainer), LargeBlockCockpitSeat (Cockpit), Conveyors
  Канал команд устройств `se.{player_id}.commands.device.{block_id}` принимает команду (возвращает 1), но состояние блока не меняется. **Блоки Merge — наиболее критичное ограничение** — их нельзя переключать программно.
- **`dev.is_enabled` — это МЕТОД, а не свойство.** Всегда вызывайте как `dev.is_enabled` (это `@property`, скобки не нужны). Самый безопасный паттерн: `enabled = "ON" if dev.is_enabled else "OFF"`.
- **`dev.telemetry` может быть `None` для многих типов устройств.** Батареи, солнечные панели, водородные двигатели и генераторы кислорода часто возвращают `None`, когда сервер SE ещё не заполнил Redis. **ВСЕГДА проверяйте**: `t = dev.telemetry or {}` перед вызовом `t.get(...)`. Без проверки получите `AttributeError: 'NoneType' object has no attribute 'get'`.
- **GenericDevice — универсальный тип.** Если `type(dev).__name__ == 'GenericDevice'`, проверьте `dev.device_type` для реального SE типа (например 'survivalkit', 'solarpanel'). У них НЕТ специфических методов — только базовые `enable()`/`disable()`.
- **ConnectorDevice имеет методы управления блокировкой/подключением.** Помимо `enable()`/`disable()`:
  - `set_state(locked=None, enabled=None)` — отправляет команду `connector_state`
  - `toggle_connect()` — переключает состояние подключения
  - `connect()` / `disconnect()` — явное управление подключением
  - `nearbyConnectors` в телеметрии — список коннекторов в радиусе стыковки
  - `connectorStatus` в телеметрии — "Connected" или "Unconnected"
  - `otherConnectorId` / `otherConnectorName` / `otherConnectorGridId` — информация о подключённой стороне

## Блоки (Blocks)

- **`g.blocks` — это dict** (block_id → BlockInfo), а не список. Не итерируйте напрямую — используйте `g.find_devices_containers()` или `g.devices.values()`.
- **`local_pos = Min * 2.5` (точно, для большой сетки).** `local_position` из `BlockInfo` — это угол Min в метрах. `Min = round(local_pos / 2.5)` для каждой оси. Проверено: Refinery при Min=(2,0,0) → local_pos=(5.0, 0.0, 0.0); Battery при Min=(-1,0,0) → local_pos=(-2.5, 0.0, 0.0).
- **API-методы могут возвращать кортежи, строки или int — не всегда объекты.** Перед вызовом `.get()`, `.name`, `.text` на возвращаемом значении проверяйте тип с помощью `type(x)`. Всегда проверяйте `isinstance(x, dict)` перед вызовом `.get()`.

## Инвентарь

- **Пустые инвентари — норма.** Всегда проверяйте `inv.items` — многие устройства сообщают об инвентаре, но он `None` или пустой список. Не предполагайте наличие ресурсов.
- **Пустые инвентари — норма на новых/неиспользуемых базах.** Не сообщайте об "ошибке" — сообщите "пусто" и отметьте как проблему ресурсов. База с полным оборудованием, но нулевыми ресурсами — обычное дело.
- **Передачи через конвейер** требуют физической стыковки корабля. `prepare_grid()` не выполняет автоматическую стыковку — сначала проверьте состояние коннектора.

## Чертежи и проекции

- **Выравнивание проекции — для каждой сетки и нетривиально.** Правильные offset/rotation зависят от мировой позиции и ориентации сетки (кватернион). Нет формулы — требуется полный перебор. Значения различаются для каждой сетки. **НИКОГДА не угадывайте значения offset** — неправильное выравнивание = сварщик строит блоки везде.
- **Сварщик ДОЛЖЕН быть отключён перед загрузкой проекций.** Nanobot BARS начинает сварку немедленно при появлении проекции. Если проекция невыровнена, она испортит базу до того, как вы успеете среагировать.
- **XML чертежей может содержать блоки без `<Min>`.** Блоки брони (1x1x1) могут опускать `<Min>`, по умолчанию позиция (0,0,0). Это нормально.
- **`request_grid_blueprint(include_connected=False)`** — всё равно может возвращать блоки из подключённых сеток, если они разделяют физическое соединение (через Connector). Проверяйте `bp['gridCount']` и количество блоков.
- **Раздувание XML чертежей ломает выравнивание.** Данные ComponentContainer/inventory могут раздувать экспорты с ~40КБ до ~800КБ. Загрузка раздутого XML вызывает `remainingBlocks == totalBlocks` для ВСЕХ смещений. **Всегда очищайте несущественные XML-теги перед загрузкой.**
- **`set_offset()` / `set_rotation()` НЕ меняют проекцию после `load_blueprint_xml()`.** Телеметрия показывает новые значения, но `remainingBlocks` не пересчитывается. Встраивайте offset/rotation в блок проектора в XML перед загрузкой.
- **Проектор может ЗАМЁРЗТЬ после быстрых команд.** После ~30+ быстрых вызовов `set_offset()`/`set_rotation()` (например при полном переборе) проектор входит в застрявшее состояние. **Восстановление: перезапуск сервера SE или плагина.**
- **`request_grid_blueprint` перестаёт работать, когда проектор заморожен.** Та же перегрузка быстрыми командами ломает и экспорт чертежей. Перезапуск исправляет обе проблемы.
- **`buildableBlocks == 0` может означать не только невыровненность.** Спроецированные блоки могут "плавать в воздухе" — не прилегать ни к какому существующему блоку. Это проблема топологии, а не выравнивания.
- **BlockOrientation должен совпадать ТОЧНО при добавлении блоков в XML чертежей.** Единственный надёжный способ получить правильную ориентацию: (а) попросить пользователя разместить блок вручную в игре, (б) экспортировать чертеж, (в) скопировать `<BlockOrientation>` из экспорта.
- **`isProjecting=False` с `remainingBlocks=0` — это НОРМАЛЬНО.** Когда все спроецированные блоки уже существуют на сетке, проектор автоматически отключает проекцию.
- **LargeAssembler — 3×3×3 блока.** При размещении в XML чертежей позиция `<Min>` определяет один угол. Проверьте, что объём 3×3×3 не перекрывает существующие блоки.
- **`proj.update()` должен вызываться после изменения offset/rotation** для отображения нового значения `remainingBlocks` в телеметрии.
- **Экспорт может содержать дублирующиеся блоки.** `blueprint_snapshot()['xml']` может включать один блок дважды. При парсинге фильтруйте только прямых детей `<CubeBlocks>`.
- **`iter('MyObjectBuilder_CubeBlock')` по XML чертежей может завышать подсчёт.** Используйте прямых детей `CubeBlocks`: `cube_blocks = root.find('.//CubeBlocks'); blocks = list(cube_blocks)`.

## Бур Nanobot

- **`set_work_mode()` имеет перепутанные значения drill/collect** (начиная с v0.3.0). `set_work_mode("drill")` отправляет значение 1 (=Collect). Используйте сырой `send_command` с `value: 2` для режима Drill. Смотрите [nanobot-drill-debugging.md](nanobot-drill-debugging.md).
- **`ScriptControlled=True` предотвращает автоматическую добычу.** Когда `ScriptControlled=True`, бур сообщает о целях через телеметрию, но `CurrentDrillTarget` остаётся `None` — добыча не происходит. **Всегда устанавливайте `ScriptControlled=False` для автоматической добычи руды.**
- **`turn_on()` НЕДОСТАТОЧНО — `start_drilling()` ОБЯЗАТЕЛЕН.** Вызов `turn_on()` включает бур, но он остаётся в режиме ожидания. Необходимо явно вызвать `start_drilling()` после `turn_on()`. Правильная последовательность: `set_property("ScriptControlled", False)` → `turn_on()` → `start_drilling()`.
- **`set_property("AreaWidth")` не меняет размер зоны.** Возвращает успех, но значение остаётся 75. Только действия `AreaWidth_Increase` / `AreaHeight_Increase` / `AreaDepth_Increase` работают (инкрементальные шаги).
- **Бур Nanobot автоматически отключается в космосе.** Периодически устанавливает `enabled=False` сам — вероятно, из-за управления питанием. Включите заново: `drill.send_command({"cmd": "set", "payload": {"property": "OnOff", "value": True}})`.
- **Зона бура — 75×75×75м (большая сетка), а не 25×25×25м.** Значительно больше, чем задокументировано для планетарных буров.
- **Смещение зоны бура в локальных координатах корабля.** Вектор гравитации из телеметрии RC должен быть спроецирован на оси ориентации корабля.
- **Добытая руда идёт в CargoContainer через конвейер, а НЕ в инвентарь бура.** С `set_use_conveyor(True)` собственный инвентарь бура показывает 0 предметов. Проверяйте инвентари `ContainerDevice`.
- **Повреждение состояния бура после множества изменений конфигурации.** Полный сброс: stop_drilling → Off → сбросить смещения на 0 → WorkMode=2 → ScriptControlled=False → turn_on → start_drilling.
- **Большие значения AreaOffset дают 0 целей.** Смещение 50+ метров к руде даёт 0 целей. Мод может ограничивать большие смещения. Используйте нулевое смещение — зона бура 75м достигает руды на расстоянии 42-46м от блока без смещения.

## Детектор руды и радар

- **`send_command` с `set` может случайно переключить флаги сканирования OreDetector.** Отправка любой команды `set` может переключить `scan.includeVoxels` с `True` на `False`. **Нельзя вернуть обратно через API.** Исправление требует **терминала в игре**. Избегайте отправки команд `set` к OreDetector без крайней необходимости.
- **`monitor_ore()` блокирует навсегда** — входит в долгий цикл. НИКОГДА не вызывайте из `execute_code`. Используйте `ore_cells()`, `contacts()` или `scan_and_wait(timeout=N)`.
- **Обнаружение рудных вокселей требует физической близости.** `detectedores` и `ore_cells()` возвращают пустой результат, даже с `includeVoxels=True`, если сетка недостаточно близко к воксельным поверхностям.
- **`asteroidIndex` и `detectedores` — разные системы.** Команда `asteroids` (через `send_command`) находит ПОЗИЦИИ астероидов в радиусе 50км. `detectedores` в телеметрии показывает МЕСТОРОЖДЕНИЯ РУДЫ в физическом радиусе детектора. Не путайте.
- **oreCells (ячейки руды) ПЕРЕДАЮТСЯ через Redis (начиная с v0.3.1+).** Ключ телеметрии `radar` теперь содержит `oreCells`. Раньше (до v0.3.1) `oreCells` и `radar` были всегда пусты.
- **`asteroidIndex` заполняется автоматически в телеметрии детектора руды.** Явная команда `asteroids` не нужна — данные появляются автоматически с `ready=True`, `count` и полным списком `items`.
- **Ore Detector `showOnScreen`/`showInTerminal` нельзя установить через API.** Отправка возвращает успех, но не меняет значение. Переключать в игре.
- **РадарController: два конфига сканирования.** (a) `bbox 100x100x5000, cell_size=10` = 1M+ тайлов → сбрасывается на 0-1.5%, НИКОГДА не используйте. (b) Узкий луч: `bbox 20x20x100, cell_size=10` = 4000 тайлов, ~5с — обнаружение на ближней дистанции. (c) Дальнего действия: `bbox 100x100x100, cell_size=100, radius=5000` = ~1000 тайлов, ~0.3с — обнаружение астероидов на 3-5км.
- **ForwardScanner: два подхода к остановке.** (A) **На основе вокселей (ПРЕДПОЧТИТЕЛЬНО)**: летите к центру астероида, `arrived = nearest < stop_distance`. (B) **Геометрический (устаревший)**: вычисляйте точку подхода, сравнивайте `nearest < OBSTACLE_RANGE AND nearest < target_distance`.
- **`solidPoints` — МИРОВЫЕ координаты, а не индексы сетки.** Каждая точка в списке `solid` из `scan_voxels()` — `[world_x, world_y, world_z]`. Для вычисления расстояния от корабля вычитайте позицию корабля.
- **`fly_to_point()` `arrival_distance=50` вызывает преждевременные остановки.** Для точных подходов используйте `arrival_distance=5.0` или `arrival_distance=10.0`. Для остановки на основе вокселей используйте `arrival_distance=9999`.
- **ForwardScanner должен напрямую `rc.disable()` при обнаружении препятствия.** Не полагайтесь только на `cancel_check` — поток сканнера должен немедленно вызвать `rc.disable()` + `rc.dampeners_on()`.
- **`compute_approach_point` должен возвращать конечную позицию остановки от ЦЕНТРА.** Правильная формула: `stop_radius = radius + stop_distance; point = center + direction * stop_radius`.
- **Ориентация гироскопа: используйте только P-контроллер, не PD.** Добавление D-члена вызывает дикую осцилляцию — корабль крутится непрерывно.

## Навигация и полёт

- **`get_world_position(device)` работает с любым BaseDevice** (CockpitDevice, RemoteControlDevice и т.д.). Предпочтительнее ручного парсинга `telemetry.get('planetPosition')`.

## Redis и окружение

- **Подключение Redis: всегда загружайте из `/workspace/.env` через dotenv.** Никогда не хардкодьте учётные данные Redis:
  ```python
  from dotenv import load_dotenv
  load_dotenv('/workspace/.env')
  ```
- **Execute_code sandbox не загружает .env** — нужно устанавливать `os.environ` явно.
- **Таймаут Redis.** Если `prepare_grid()` зависает, Redis может быть недоступен. Проверьте с помощью `socket.create_connection((host, 6379), timeout=5)`.
- **Redis ACL ограничивает KEYS и SCAN.** Пользователь secontrol Redis не может выполнять команды `KEYS` или `SCAN` — `NoPermissionError`. Используйте прямой `GET` с известными паттернами ключей.
- **`se:{owner_id}:grids` возвращает None → сервер SE офлайн.** Когда ключ сеток пуст и известные ключи телеметрии тоже возвращают None, сервер Space Engineers не запущен или Redis-мост отключён.
- **Путь HOME различается между контекстами.** Интерактивные сессии используют `/home/hermeswebui`, cron/запланированные задачи — `/home/hermes`. Всегда проверяйте `os.environ['HOME']` и используйте абсолютные пути.
- **Cron-скрипты, вызывающие `hermes`, должны использовать правильный путь к бинарнику.** Обёртка `~/.hermes/hermes-agent/hermes` может иметь сломанный shebang. Рабочий бинарник обычно находится в `/app/venv/bin/hermes`.

## Git и pip

- **Inline f-strings с кавычками в execute_code** вызывают SyntaxError. Записывайте во временный файл (`<tmp_dir>/script.py`, runtime path) и запускайте через subprocess.
- **`/opt/hermes/.venv/bin/python` может не иметь модуля `pip`.** Используйте `subprocess.run(['pip', 'install', ...])` с системным pip, или `subprocess.run([sys.executable, '-m', 'ensurepip'])` сначала.
- **Приоритет пути импорта: `/workspace/src` на sys.path скрывает пакет, установленный через pip.** Если существуют оба — `/workspace/src/secontrol/` и установленный через pip secontrol, добавление `sys.path.insert(0, '/workspace/src')` заставит Python импортировать СТАРЫЙ исходник. Исправление: не добавляйте `/workspace/src` в sys.path при использовании `pip install -e /tmp/secontrol-fresh`.
- **`pip install -e .` может завершиться ошибкой из-за прав на egg-info.** Обходной путь: используйте `PYTHONPATH=src` вместо установки, или `rm -rf src/secontrol.egg-info` и повторите.
- **Исчерпание дискового пространства вызывает каскадные сбои.** Когда `/tmp` заполняется директориями `hermes_sandbox_*`. Исправление: `rm -rf /tmp/hermes_sandbox_*`.
- **Файлы Git-репозитория принадлежат другому UID — `pip install -e` из свежего клона как обходной путь.** Рабочий путь: `git clone --depth=1 https://github.com/rootfabric/secontrol.git /tmp/secontrol-fresh && pip install -e /tmp/secontrol-fresh`.
- **`git pull` завершается ошибкой при неотслеживаемых файлах примеров.** Исправление: `rm <конфликтный-файл> && git pull origin main`.
- **Процедура обновления secontrol.** `cd /workspace && git pull origin main` → `pip install -e /workspace`. Если pull завершается ошибкой из-за неотслеживаемых файлов, удалите их сначала.

## HSV и покраска

- **Нормализация HSV для `paint_block` — используйте `hsv=[H, S/100, V/100]`** для соответствия цветам шлифовки в игре. Функция `_normalize_hsv_triplet()`: H > 1 → /360 (градусы в 0-1), S/V через `_normalize_unit()`: 0→0.0, 0-1→как есть, 1-100→/100, 100-255→/255. SE ColorMaskHSV: x=H(0-1), y=S(-1..1), z=V(-1..1).
- **Режим разборки Nanobot BARS** работает по совпадению цвета — покрасьте блоки, настройте цвет разборки в CustomData, затем включите сварщик. Не забудьте отключить после.
