# Приоритет очистителей

Основной операторский скрипт:

```powershell
python examples/organized/refinery/refinery_priority_operator.py --grid farpost0 --evaluate
```

Конфиг порядка руд:

```text
examples/organized/refinery/refinery_priority_config.json
```

## Режим оценки

Ничего не меняет, только показывает руду на гриде, желаемую очередь и действия:

```powershell
python examples/organized/refinery/refinery_priority_operator.py --grid farpost0 --evaluate
```

Если не указать `--apply`, скрипт сам работает в безопасном режиме оценки.

## Применить приоритет

```powershell
python examples/organized/refinery/refinery_priority_operator.py --grid farpost0 --apply
```

Скрипт смотрит все очистители на гриде, строит желаемую очередь по `ore_order`, переставляет очередь очистителя и перекладывает нужную руду во входной инвентарь.

## Запуск циклом

```powershell
python examples/organized/refinery/refinery_priority_operator.py --grid farpost0 --apply --loop --interval 30
```

## Учитывать очередь сборки

```powershell
python examples/organized/refinery/refinery_priority_operator.py --grid farpost0 --apply --from-assembler-queue
```

В этом режиме скрипт смотрит очередь ассемблеров, берёт prerequisites из blueprint-ов, считает нужные слитки и поднимает руду для дефицитных слитков вверх приоритета.

Например, если в очереди сборки есть компоненты, которым нужны урановые или платиновые слитки, а этих слитков мало, `Uranium` или `Platinum` временно попадут в начало фактического приоритета.

## Настройка порядка руд

В `refinery_priority_config.json` меняется список:

```json
"ore_order": [
  "Uranium",
  "Platinum",
  "Gold",
  "Silver",
  "Cobalt",
  "Magnesium",
  "Nickel",
  "Silicon",
  "Iron",
  "Stone"
]
```

Первое место — самый высокий приоритет.

## Контейнеры-источники

По умолчанию скрипт сначала ищет контейнеры с тегами:

```json
"source_container_tags": ["source", "input", "ore"]
```

Теги можно писать в имени блока, например `[ore]`, `[source]`, или в Custom Data. Если таких контейнеров нет, используется любой обычный контейнер на гриде, потому что включено:

```json
"use_all_containers_if_no_tagged_sources": true
```

## Безопасность

Скрипт не удаляет руду, а только переносит её между инвентарями. Перед первым применением обязательно проверь план через `--evaluate`.

## Почему может казаться, что приоритет не сработал

Space Engineers плавит то, что стоит первым во входном инвентаре очистителя, а очередь production иногда не переупорядочивает уже лежащие стеки руды. Поэтому одного `queue_clear/queue_add` недостаточно: если в первом слоте лежит `Silver`, а `Uranium` уже лежит вторым стеком, игра продолжит показывать и часто обрабатывать серебро первым.

Исправленная версия делает отдельную операцию после перекладки руды:

```text
Uranium -> targetSlotId=0 во входном инвентаре очистителя
```

За это отвечает настройка:

```json
"force_priority_ore_to_first_input_slot": true
```

Обычно её надо оставить включённой. Если нужно только переписывать очередь без перестановки первого слота входного инвентаря, можно поставить `false`.

## Если руда не переставляется в первый слот

В некоторых версиях плагина команда перестановки предмета внутри того же inputInventory может формально проходить, но фактически не менять первый слот, если он занят другой рудой. Поэтому операторский скрипт теперь использует обходной алгоритм:

1. временно выносит руду, которая стоит перед приоритетной;
2. ждёт, пока приоритетная руда станет первым стеком;
3. возвращает вынесенную руду назад после приоритетной.

За это отвечает настройка:

```json
"restore_displaced_lower_priority_ore": true
```

Если поставить `false`, нижний приоритет будет вынесен из очистителя и не вернётся автоматически.

## Проверка фактического переноса

Новая версия больше не верит одному факту отправки Redis-команды. После каждой операции переноса она перечитывает телеметрию и проверяет, что входной инвентарь очистителя реально изменился.

Для операций с `inputInventory` очистителя команда теперь отправляется через сам очиститель, даже если источник руды — cargo-контейнер. Это важно из-за особенности текущего C#-плагина: `fromInventoryIndex`/`toInventoryIndex` гарантированно учитываются только для устройства, которое обрабатывает команду. Если отправить команду с cargo-контейнера в refinery, индекс удалённого refinery может быть проигнорирован.

Если временный контейнер выбран неправильно, в логе теперь будет видно его имя или fallback-метка вида:

```text
container(123456789)
```

Для временного выноса нижнего приоритета лучше иметь cargo-контейнер с тегом в имени:

```text
[ore]
```

или:

```text
[buffer]
```

Список тегов для временного буфера задаётся в конфиге:

```json
"temporary_storage_tags": ["ore", "cargo", "storage", "buffer", "input"]
```

Таймаут подтверждения переноса:

```json
"verify_transfer_timeout": 2.0
```

Если в логе написано:

```text
не смог вынести Silver; скорее всего transfer_items не попал в inputInventory
```

значит Python отправил несколько вариантов команды, но телеметрия входа очистителя не изменилась. Тогда нужно смотреть лог плагина по строкам `ContainerDevice validation` и `item_transfer_failed`.

## Проверка версии

Новая версия при запуске печатает строку:

```text
Скрипт: refinery-priority-v5-keep-lower-priority-after-first-2026-05-31
```

Если этой строки нет, значит Windows запускает старый файл и исправление не применилось.

По умолчанию временно вынесенная нижнеприоритетная руда не возвращается обратно во вход очистителя, чтобы она снова не встала перед Uranium. Вернуть старое поведение можно флагом:

```powershell
python examples/organized/refinery/refinery_priority_operator.py --grid farpost0 --apply --restore-displaced
```

## Поведение v5

Версия v5 не оставляет нижнеприоритетную руду навсегда в cargo-контейнере.
Если во входе было `Cobalt, Uranium`, скрипт делает рабочую перестановку:

```text
1. временно выносит Cobalt из inputInventory;
2. проверяет, что Uranium стал первым;
3. возвращает/добирает Cobalt обратно уже после Uranium через общий этап заполнения входа.
```

Ожидаемый результат после применения:

```text
вход сейчас: Uranium, Cobalt
```

Если после применения виден только `Uranium`, запусти `--apply` ещё раз: v5 считает неполный вход `Uranium` как `input_ok=False` и должен добрать `Cobalt` из контейнера.
